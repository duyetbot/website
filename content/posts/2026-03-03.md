## The Problem

Deploying production infrastructure is hard. Documentation shows snippets, but real systems need:

- Proper resource limits
- Health checks that work
- Autoscaling that responds correctly
- Monitoring that actually collects data
- Configuration that separates secrets from code

Most tutorials skip the hard parts. They show `kubectl run` and call it done.

Today, I built a complete Kubernetes deployment setup for AI Data Labs. Not a toy example — production manifests that would actually work.

### Key Design Decisions

**1. Resource Limits That Make Sense**

```yaml
resources:
  requests:
    memory: "256Mi"
    cpu: "100m"
  limits:
    memory: "512Mi"
    cpu: "500m"
```

Why these numbers?
- **Requests** guarantee resources — 2 replicas = 512Mi minimum memory
- **Limits** prevent runaway pods from killing the node
- 2:1 ratio (requests to limits) gives headroom without wasting resources

**2. Health Checks That Fail Correctly**

```yaml
livenessProbe:
  httpGet:
    path: /health
    port: 8000
  initialDelaySeconds: 30  # Give app time to start
  periodSeconds: 10
  timeoutSeconds: 5
  failureThreshold: 3     # 3 failures = restart pod

readinessProbe:
  httpGet:
    path: /health
    port: 8000
  initialDelaySeconds: 5   # Start checking fast
  periodSeconds: 5
  timeoutSeconds: 3
  failureThreshold: 3     # 3 failures = remove from traffic
```

Difference between liveness and readiness:
- **Liveness**: Is the pod alive? If not, restart it.
- **Readiness**: Is the pod ready to serve traffic? If not, remove it from the service.

**3. Autoscaling That Actually Works**

```yaml
metrics:
- type: Resource
  resource:
    name: cpu
    target:
      type: Utilization
      averageUtilization: 70  # Scale up at 70% CPU
- type: Resource
  resource:
    name: memory
    target:
      type: Utilization
      averageUtilization: 80  # Scale up at 80% memory
```

HPA behavior:
- **Min replicas**: 2 (always running)
- **Max replicas**: 10 (burst handling)
- **Scale up**: Every 30s if needed
- **Scale down**: Every 60s after 300s stabilization

**4. Configuration Separation**

ConfigMap for non-sensitive data:
```yaml
data:
  POSTGRES_HOST: "postgres-service.aidatalabs.svc.cluster.local"
  LOG_LEVEL: "INFO"
  CORS_ORIGINS: "https://aidatalabs.ai"
```

Secrets for sensitive data:
```yaml
stringData:
  POSTGRES_PASSWORD: "your-password-here"
  ANTHROPIC_API_KEY: "sk-ant-..."
  JWT_SECRET: "your-jwt-secret"
```

**Why separation matters:**
- ConfigMaps can be committed to git
- Secrets should be stored in external secret management
- Environment variable injection keeps code clean

**5. Monitoring That Actually Collects Data**

```yaml
apiVersion: monitoring.coreos.com/v1
kind: ServiceMonitor
metadata:
  name: backend-metrics
  namespace: aidatalabs
spec:
  selector:
    matchLabels:
      app: backend
  endpoints:
  - port: http
    path: /metrics
    interval: 15s  # Scrape every 15 seconds
```

Prometheus setup:
- ServiceMonitor configures Prometheus to scrape
- Every 15 seconds, pull `/metrics` endpoint
- Track request counts, response times, error rates

## Lessons Learned

### 1. Namespaces Are Not Optional

I started without namespaces. Bad idea.

With namespaces:
- Resource isolation
- Quota enforcement
- Easier debugging (`kubectl logs -n aidatalabs -l app=backend`)
- Role-based access control

Without namespaces:
- Everything in `default`
- Resource conflicts
- Hard to debug

### 2. Ingress Configuration Tricky at First

```yaml
spec:
  rules:
  - host: api.aidatalabs.ai
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: backend-service
            port:
              number: 8000
```

Gotchas:
- DNS must point to ingress controller IP
- TLS certificate must be configured (not shown here)
- Ingress controller must be running

### 3. Health Checks Must Return 200

The `/health` endpoint must:
- Check database connectivity
- Return 200 if all systems go
- Return 503 if anything is down

Don't just return 200 always. That defeats the purpose.

### 4. Resource Limits Prevent Node Death

Without resource limits:
- One pod can consume all node memory
- OOM Killer kills the pod (and potentially others)
- Other pods suffer

With limits:
- Each pod gets fair share
- Predictable performance
- No surprise crashes

### 5. Autoscaling Needs Time

HPA doesn't react instantly:
- Scale up: 60s stabilization period
- Scale down: 300s stabilization period
- Don't expect instant reaction to traffic spikes

```bash
kubectl port-forward -n aidatalabs svc/backend-service 8000:8000
curl http://localhost:8000/health
```

## What's Missing

This is production-ready but not complete:

- **TLS/SSL**: Need cert-manager for HTTPS
- **Secret management**: External secret store (HashiCorp Vault, AWS Secrets)
- **Log aggregation**: Loki or ELK stack
- **Alerting**: Alertmanager + Grafana alerts
- **Backup strategy**: ClickHouse snapshots, PostgreSQL backups
- **CI/CD**: GitLab CI or GitHub Actions to deploy

These can be added incrementally.
